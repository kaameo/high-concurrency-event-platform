# ADR-010: 클라우드 Kubernetes 환경 선택 — AWS EKS

## 상태

Proposed

## 날짜

2026-04-09

## 컨텍스트

Phase 3~4 실험에서 로컬 kind 클러스터의 구조적 한계가 확인되었다:

1. **단일 Kafka 브로커**: 파티션 병렬화 효과 미확인 (실험 C), MTTR 목표 미달성 (실험 D)
2. **2 Worker 노드 제약**: 10 Pod 배치 시 물리 리소스 경합으로 실험 왜곡 (실험 B)
3. **단일 머신 I/O 공유**: 모든 컨테이너가 동일 디스크/CPU/메모리를 공유

다중 노드 분산 환경에서 재실험이 필요하며, 클라우드 Kubernetes 서비스를 선택해야 한다.

### 요구사항

- 다중 노드 (최소 3대) 분산 클러스터
- Kafka 3 브로커 운영 가능한 리소스
- 비용 최소화 (테스트 목적, 상시 운영 아님)
- Helm 기반 인프라 배포 호환
- k6 부하 테스트 실행 가능

## 검토 옵션

### Option 1: AWS EKS + Spot Instance

| 항목 | 내용 |
|------|------|
| 관리형 K8s | EKS (Control Plane 관리) |
| 노드 | EC2 Spot Instance (m5.large / m5a.large) |
| 비용 | 클러스터 $0.10/h + Spot ~$0.05/h × 3대 = ~$0.25/h |
| 장점 | 프로덕션급 환경, AWS 에코시스템, 이력서 가치 |
| 단점 | 초기 설정 복잡도 (IAM, VPC, eksctl) |

### Option 2: GKE Autopilot

| 항목 | 내용 |
|------|------|
| 관리형 K8s | GKE Autopilot (노드 관리 불필요) |
| 노드 | Google 관리 (Pod 단위 과금) |
| 비용 | 신규 계정 $300 무료 크레딧 |
| 장점 | 노드 관리 불필요, 무료 크레딧 |
| 단점 | 크레딧 소진 후 비용 급증, AWS 경험 불가 |

### Option 3: Vultr Kubernetes

| 항목 | 내용 |
|------|------|
| 관리형 K8s | Vultr Kubernetes Engine |
| 노드 | Vultr Cloud Compute |
| 비용 | ~$30-60/월 |
| 장점 | 저렴, 간단한 설정 |
| 단점 | 에코시스템 빈약, 포트폴리오 가치 낮음 |

### Option 4: Hetzner + k3s

| 항목 | 내용 |
|------|------|
| 관리형 K8s | 직접 구축 (k3s) |
| 노드 | Hetzner CAX11 ARM × 3 |
| 비용 | ~€12/월 (~$13) |
| 장점 | 최저가, 실제 분산 노드 |
| 단점 | 직접 구축/관리, ARM 호환성 확인 필요 |

### Option 5: Oracle Cloud Free Tier

| 항목 | 내용 |
|------|------|
| 관리형 K8s | OKE (Oracle Kubernetes Engine) |
| 노드 | ARM A1 (4 OCPU, 24GB) 무료 |
| 비용 | $0 |
| 장점 | 완전 무료 |
| 단점 | 단일 노드 제약, 리소스 부족 가능, 포트폴리오 가치 낮음 |

## 결정

**AWS EKS + Spot Instance를 채택한다** (Option 1).

## 근거

### 1. 포트폴리오 가치 극대화

백엔드 엔지니어 포지션에서 AWS 경험은 핵심 역량이다. "AWS EKS에서 100k TPS 부하 테스트 수행"은 이력서에서 실질적 가치를 가진다. GKE, Vultr, Hetzner 경험은 상대적으로 어필 효과가 낮다.

### 2. 비용 통제 가능

Spot Instance + 1회성 실험 패턴으로 실제 비용을 $3-5 수준으로 제한할 수 있다:

```
EKS 클러스터: $0.10/h × 3h = $0.30
Spot m5.large × 3: $0.05/h × 3 × 3h = $0.45
ECR + 데이터 전송: ~$0.50
────────────────────────────────
합계: ~$1.25 / 세션 (3회 = ~$3.75)
```

핵심 원칙: 실험할 때만 클러스터 생성, 결과 캡처 후 즉시 삭제.

### 3. 프로덕션 환경과의 유사성

EKS는 실제 프로덕션에서 가장 많이 사용되는 관리형 K8s 서비스다. kind → EKS 전환 과정 자체가 "로컬 → 클라우드 마이그레이션" 경험이 된다.

### 4. 기존 도구 체인 호환

eksctl로 클러스터 생성, Helm으로 인프라 배포, kubectl로 관리 — 기존 kind 환경과 동일한 워크플로우를 유지하면서 클라우드 환경으로 전환 가능.

### GKE를 선택하지 않은 이유

- $300 무료 크레딧은 매력적이나, AWS 경험 대비 포트폴리오 가치가 낮음
- EKS 비용이 $3-5 수준으로 충분히 낮아 무료 크레딧의 이점이 결정적이지 않음
- 단, GKE 크레딧이 이미 있다면 병행 실험으로 "멀티 클라우드 경험"을 추가할 수 있음

### Hetzner + k3s를 선택하지 않은 이유

- k3s 직접 구축은 학습 가치가 있으나, 관리형 K8s 대비 운영 오버헤드가 큼
- ARM 환경에서 Kafka, Spring Boot JVM 호환성 추가 검증 필요
- 이 프로젝트의 목표는 K8s 구축이 아닌 "고동시성 시스템 검증"

## 결과

- Phase 5 실험 계획에 따라 EKS 환경에서 실험 E/F/G를 수행
- 실험 결과를 Phase 3~4와 비교하여 "로컬 vs 클라우드" 성능 차이 문서화
- ADR-008 (HPA), ADR-009 (장애 복구)의 한계 사항을 클라우드 환경에서 재검증
